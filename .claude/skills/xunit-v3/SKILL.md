# xUnit v3 — house testing conventions

House rule: new test projects use **xUnit v3** (`xunit.v3` package) — not `xunit` 2.x, not MSTest/NUnit (deviating needs a documented compatibility reason). Test projects target `net10.0`, live under `tests/`, and are named `<ProjectUnderTest>.Tests`.

## Project setup (MTP-native — the default)

xUnit v3 test projects are **standalone executables** on Microsoft.Testing.Platform. Do NOT add the VSTest-era boilerplate (`Microsoft.NET.Test.Sdk`, `xunit.runner.visualstudio`, `coverlet.collector`).

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <!-- Required: v3 test projects are executables; Main is generated by xunit.v3.core -->
    <OutputType>Exe</OutputType>
    <LangVersion>14</LangVersion>
    <Nullable>enable</Nullable>
    <ImplicitUsings>disable</ImplicitUsings>
    <TreatWarningsAsErrors>true</TreatWarningsAsErrors>
  </PropertyGroup>
  <ItemGroup>
    <PackageReference Include="xunit.v3" Version="3.2.2" />
  </ItemGroup>
  <ItemGroup>
    <ProjectReference Include="../../src/MyLib/MyLib.csproj" />
  </ItemGroup>
</Project>
```

Version note: 3.2.2 is stable as of 2026-07 — use the latest stable **3.x**; xUnit 4.x is prerelease, don't use it.

On the .NET 10 SDK, switch `dotnet test` to MTP mode once per repo in **global.json**:

```json
{ "test": { "runner": "Microsoft.Testing.Platform" } }
```

Do not emit a `dotnet.config` (removed preview mechanism); `TestingPlatformDotnetTestSupport` is only for .NET 8/9 SDKs.

Run with `dotnet test` using MTP filters (`--filter-class`, `--filter-method`, `--filter-trait`, `--filter-query` — not VSTest's `--filter FullyQualifiedName~…`), or run the built exe directly.

## Writing tests — v3 API notes

```csharp
using Xunit;

namespace MyLib.Tests;

public sealed class CalculatorTests
{
    [Fact]
    public void Add_ReturnsSum() => Assert.Equal(5, Calculator.Add(2, 3));

    [Theory]
    [InlineData(0, 0, 0)]
    [InlineData(-1, 1, 0)]
    public void Add_VariousInputs(int a, int b, int expected)
        => Assert.Equal(expected, Calculator.Add(a, b));
}
```

- `ITestOutputHelper` lives in `Xunit` (the `Xunit.Abstractions` namespace is gone); constructor injection still works, or use `TestContext.Current.TestOutputHelper`.
- `IAsyncLifetime` now returns **`ValueTask`** from `InitializeAsync`/`DisposeAsync` and inherits `IAsyncDisposable`.
- Pass `TestContext.Current.CancellationToken` into async code under test.
- Dynamic skips: `Assert.Skip(…)`, `Assert.SkipWhen(…)`, `Assert.SkipUnless(…)`.
- No `async void` tests (they fast-fail at runtime) — return `Task`/`ValueTask`.
- `xunit.runner.json` still works: `<Content Include="xunit.runner.json" CopyToOutputDirectory="PreserveNewest" />`.

## Common stale v2-isms

| Wrong (v2 habit) | Right (v3) |
|---|---|
| `<PackageReference Include="xunit" Version="2.x" />` | `xunit.v3` |
| `Microsoft.NET.Test.Sdk` + `xunit.runner.visualstudio` + coverlet boilerplate | just `xunit.v3` + the global.json test runner entry |
| Missing `<OutputType>Exe</OutputType>` or a hand-written `Program.Main` | Exe output; `Main` is generated |
| `using Xunit.Abstractions;` | `using Xunit;` |
| `Task IAsyncLifetime.InitializeAsync()` | `ValueTask` |

If an existing solution is on xUnit v2 or VSTest mode, stay consistent within it and flag the divergence — migrating is its own work item, not a drive-by.
